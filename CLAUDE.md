# ShepherdLog вЂ” CLAUDE.md
*Pastoral Care Reporting Platform*

---

## What This Is

ShepherdLog is a web-based pastoral care reporting platform for a three-tier disciple-making ministry. Leaders submit structured reports on congregation members up to the pastor. A separate follow-up team logs calls to first-time visitors. The pastor has full read access across all reporting streams.

**Core constraint:** No congregation member or first-timer has a platform account. Only pastor, leaders, and follow-up team members authenticate.

## Auth Architecture

- No self-registration. All accounts created by the pastor via the `/users` API endpoint.
- On login, the server returns a signed JWT containing `{ userId, role }`.
- The client stores the JWT in `localStorage` under the key `sl_token`.
- Every API request includes `Authorization: Bearer <token>` in the header.
- The `authenticate` middleware verifies the token and attaches `req.user = { id, role }` to every protected request.
- The `requireRole(...roles)` middleware gates routes to specific roles.
- JWT expiry is 7 days. No refresh token вЂ” on expiry, user is redirected to `/login`.

### Role values
`pastor` | `leader` | `followup_team_lead` | `followup_team_member`

## API Design

- All routes prefixed `/api/v1/`
- JSON request and response bodies throughout
- Errors return `{ error: string, code?: string }` with appropriate HTTP status
- Successful mutations return the created or updated entity
- List endpoints return `{ data: T[], total: number }` for paginatable resources

### Access control pattern

Every route applies middleware in this order:
1. `authenticate` вЂ” verify JWT
2. `requireRole('pastor', 'leader')` вЂ” check role (or whatever roles apply)
3. Route handler вЂ” applies additional scoping (e.g. a leader can only see their own members)

Access control is **never** delegated to the frontend alone. The API enforces role scoping on every query.

## Key Conventions

### Prisma
- Use the singleton from `lib/prisma.ts` вЂ” never instantiate PrismaClient directly in a route or service
- All database operations go through service files вЂ” routes call services, services call Prisma
- Use Prisma transactions (`prisma.$transaction`) for any multi-table write (e.g. member conversion)

### Activity Logging
- `activityLogService.writeLog()` is called inside the same service function as the triggering operation вЂ” never fire-and-forget, never called from a route handler directly
- Activity log rows are never deleted вЂ” no delete endpoint exists for activity logs

### Confidentiality
- `isConfidential = true` reports: the Prisma query in `members.service.ts` must explicitly filter these out for leader reads. Do not rely on any frontend filtering.
- A leader may only see a confidential report they wrote themselves, plus all non-confidential reports on their assigned members

### Safety Flags
- `isSafetyFlagged = true` reports must be excluded from all delete and redact operations at the service layer вЂ” check the flag before proceeding, return 403 if flagged
- A safety-flagged report submission calls `notificationsService.createNotification()` and `notificationsService.sendEmail()` for the pastor immediately вЂ” synchronously before the response is returned

### Silence Detection
- `lastReportDate` on each member is updated inside the `createMemberReport` service function (Prisma `update` inside the same transaction as the report insert)
- Silence status is computed in the service layer: `now - lastReportDate > settings.reportThresholdDays` в†’ overdue; `> 2 Г— threshold` в†’ significantly overdue
- Never compute silence on the frontend

### Scheduled Jobs (`server/src/jobs/scheduler.ts`)
- `node-cron` runs inside the Express process
- Two jobs:
  - **Report reminders** вЂ” runs on the configured day of week, checks each leader for members with no report this cycle, sends email + creates notification
  - **Silence detection** вЂ” runs daily at 07:00 WAT (07:00 Africa/Accra = 07:00 UTC), queries all members past threshold, creates pastor notifications if not already notified today

## Absolute Rules

- Do not implement self-registration
- Do not allow members or first-timers to authenticate
- Do not create a delete endpoint for activity logs
- Do not allow delete or redact operations on `isSafetyFlagged = true` reports вЂ” enforce in the service layer
- Do not trust the frontend for access scoping вЂ” always filter by `req.user.id` and `req.user.role` in the service
- The "Convert to Son/Daughter" action must use a Prisma transaction вЂ” the member insert and first-timer status update must either both succeed or both fail
- Never expose `JWT_SECRET` or `DATABASE_URL` to the client
- A leader must never be able to retrieve another leader's members or reports through any API endpoint, regardless of query parameters passed
